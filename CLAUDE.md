---

## 1. Контекст проекта (чтобы не путаться)

- **Что это:** веб-чекер аккаунтов Facebook + генератор 2FA-кодов. Бэкенд —
  Node.js/Express (`server.js`, ESM, `"type": "module"`). Фронтенд — ванильный
  JS/HTML/CSS в `public/` (без сборки и фреймворков).
- **Checker upstream:** `https://check.fb.tools/api/check/facebook` —
  **публичный, без авторизации**. Тело: `{ inputData: [<fbid|url|cookies>...] }`.
  Ответ: `data[].{account,uid,status:{name,message},origin,...}` + `info`.
  Статусы `status.name`: `valid` / `invalid` / `noexist` / `error`.
  ⚠️ Маппить результат по `origin` (исходная строка), а не по `account`/`uid`
  (для URL/cookies они отличаются). Переопределяется env `CHECK_URL`.
  ⚠️ Старого NPPR (`npprservices.pro`, `NPPR_TOKEN`) в коде больше НЕТ.
- **2FA:** TOTP по RFC 6238 (SHA-1, 6 цифр, 30 с) на встроенном `node:crypto`,
  без зависимостей. На фронте (`public/2fa.html` + `2fa.js`) расчёт клиентский
  через Web Crypto — секрет не покидает браузер.
- **Конфиг:** токены не нужны. Опционально `PORT`, `CHECK_URL`, `NODE_VERSION`,
  `ALLOWED_ORIGINS` (CORS allowlist через запятую; пусто → всё разрешено),
  `RATE_LIMIT_MAX` / `RATE_LIMIT_WINDOW_MS` (по умолчанию 60 / 60000, per-IP на
  `/api/*` кроме `/api/ping`).
- **Endpoints:** `/api/ping`, `/api/check`, `/api/check/stream` (основной, SSE),
  `/api/get_uid/:id` (legacy), `/api/otp/:secret` (2FA → `{ok,data:{otp,timeRemaining}}`).
  SSE-`batch` отдаёт `{line, id, uid, profileLink, valid, status}`.
- **Фронт Checker:** шлёт строку, если в ней есть FB ID ИЛИ `facebook.com/` /
  `instagram.com/` / cookies (`c_user=`) — username-URL тоже проверяются. Дедуп
  по FB ID, иначе по самой строке.
- **Константы апстрима** (`server.js`): batch 50, retry 1, delay 1.5 c,
  timeout 30 c, `STREAM_UPSTREAM_CONCURRENCY` = 4.
- **Деплой:** два таргета — Render (`render.yaml`, бэкенд+статика) и GitHub
  Pages (`.github/workflows/pages.yml`, только `public/`). На `*.github.io`
  фронт ходит на Render через `public/config.js` (`PROXY_BASE`).
- **Запуск:** `npm run dev` (dev), `npm run start` (prod).

## 2. Обязательные скилы (ALWAYS)

Подключай скилы ДО действия, а не после. Это не опционально:

- **`superpowers:brainstorming`** — ВСЕГДА перед новой фичей или изменением
  поведения. Сначала дизайн и согласование, потом код.
- **`superpowers:test-driven-development`** — ВСЕГДА при реализации фичи/багфикса
  (где это применимо к коду, а не к правке доков).
- **`superpowers:systematic-debugging`** — ВСЕГДА при любом баге, падении или
  неожиданном поведении, ДО предложения фикса.
- **`superpowers:verification-before-completion`** — ВСЕГДА перед тем как сказать
  «готово».
- **`/code-review`** — перед коммитом любых изменений кода.
- **`/security-review`** — при изменениях в `server.js`, обработке ввода, CORS
  или сетевых вызовах к upstream.
- **`ui-ux-pro-max`** или **`frontend-design`** — при работе с `public/` (вёрстка,
  стили, UX).

Если есть хоть 1% вероятности, что скил применим — подключай его.

## 3. Самопроверка после изменений (MUST)

В проекте нет тест-раннера. Поэтому после ЛЮБОЙ правки кода прогоняй
самопроверку вручную и показывай результат:

1. **Синтаксис:** `node --check server.js`.
2. **Поднять и дёрнуть:** запусти `npm run dev`, затем проверь живость:
   - `curl -s localhost:3000/api/ping` → ожидается `{ ok: true, ... }`;
   - прогон тестовой строки через `/api/check` и проверка, что форма ответа
     осталась `{ valid, invalid, total, breakdown }`;
   - для правок SSE — проверить, что `/api/check/stream` шлёт события
     `start` → `batch` → `end`.
3. **Синхронность доков и кода:** после правок сверь, что `README.md`,
   `docs/` и `public/` не противоречат коду — endpoints, константы, имена
   переменных. Расхождения чини в том же заходе.
4. **Фронт:** если менял `public/`, проверь в браузере основной сценарий
   (вставка строк → «Проверить» → прогресс-бар → результаты).

Никогда не пиши «готово/работает», если самопроверка не пройдена. Если шаг
пропущен — скажи об этом прямо.

- ВСЕГДА перед задачей читай `docs/IDEAS_AND_IMPROVEMENTS.md` и
  `docs/ANTI_SLEEP_PLAN.md` и сверяй задачу с ними.

- Если пользователь даёт новую идею — соотнеси её с этими файлами и при
  необходимости предложи дописать туда.

## 6. Git и деплой (можно коммитить и пушить)

- Коммитить и пушить в `origin/main` МОЖНО без отдельного подтверждения —
  но только ПОСЛЕ успешной самопроверки (раздел 3). Не пушь сломанное.
- Пуш в `main` триггерит деплой: GitHub Pages (статика `public/`) и Render
  (бэкенд, если сервис активен).
- `node_modules/` не трекается (см. `.gitignore`). Секреты — только в `.env`,
  который в git не попадает.
- Доки держи в синхроне с кодом: меняешь поведение/endpoint/константу — правь
  README/docs/CLAUDE.md в том же изменении.
- Сообщения коммитов заканчивай строкой:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`
- Отчитывайся честно: проверка не прошла — покажи вывод; шаг пропущен — скажи.

## 7. Проверка на проде (после пуша/деплоя) — MUST

После пуша ВСЕГДА проверяй весь функционал на проде, не только локально:

- Дождись деплоя (Pages ~1–2 мин; Render ~1–3 мин, free-tier может
  «просыпаться» дольше).
- **Бэкенд (Render):** `curl <render-url>/api/ping` → `{ ok:true }`; прогон
  `/api/check` тестовой строкой (сверить статусы); `/api/otp/<secret>` сверить
  с эталоном TOTP.
- **Фронт (GitHub Pages):** через Chrome MCP открой страницу, проверь Checker
  (вставка → проверка → результаты) и 2FA (код на странице совпадает с API).
- Если Render **приостановлен/спит** — зафиксируй и сообщи пользователю: часть
  прод-проверок недоступна, пока сервис не поднят (статика на Pages при этом
  работает; 2FA-страница работает без бэкенда, Checker на Pages — нет).
